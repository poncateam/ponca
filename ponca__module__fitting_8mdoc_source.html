257 \subsubsection fitting_newestimbase Minimal requirements
258
259 In order to make an estimator compatible with Ponca Fitting API, new estimator should have the following structure:
260
261 \snippet concepts.hpp ComputationalObjectConcept
262
263 \note The macro PONCA_FITTING_DECLARE_DEFAULT_TYPES defines the `Base` type (alias for T) as well as `Scalar`, `VectorType` and `NeighborFilter` for
264 manipulation of related data. This macro is defined within &lt;Fitting/defines.h&gt;
265 \note If the estimator is expected to be the upmost parent of the hierarchy, do not inherit from T and default it to be void.
266 \warning The macro PONCA_FITTING_DECLARE_DEFAULT_TYPES opens a public section.
267
268 In order for each computation to work properly, the functions `init`, `addLocalNeighbor` and `finalize` should call their Base version (`Base::init`, `Base::addLocalNeighbor`, `Base::finalize`).
269 Presumably, this is among the first thing each of these function does.
270
271 \subsubsection fitting_newcapabilities Computational objets capabilities and requirements
272
273 Aggregating small computational objects allows to minimize code duplication, but also to easily combine different techniques.
274 For instance, GLSDer can be used independently of the fitting technique, e.g. Ponca::OrientedSphereFit or Ponca::UnorientedSphereFit, as long as the fitted \ref PrimitiveBase "Primitive" is an AlgebraicSphere.
275
276 In order to detect if the computational objects are correctly combined, Ponca provides a compile-time requirement/capability system:
277 each computational objects check if the other classes of the CRTP provides the required computations.
278 For instance, in the GLSParam class, we define the following enum:
279 \snippet{lineno} gls.h Requirements
280 `Base` is a type defined from the CRTP and that represents the inherited classes.
281 If `Base` does not define `PROVIDES_ALGEBRAIC_SPHERE`, this line generates an error when compiling the program.
282 For instance, compiling the type
283 \code
284 typedef Basket<Point, WeightSmoothFunc, GLSParam> FitSmoothOriented;
285 \endcode
286 generates the following error:
287 \code
288 Building CXX object /xxxxx/wrongType.cpp.o
289 In file included from /xxxxx/wrongType.cpp:19:
290 /xxxxx/ponca/Ponca/src/Fitting/gls.h:51:23: error: no member named 'PROVIDES_ALGEBRAIC_SPHERE' in 'Ponca::PrimitiveBase<PointPositionNormal<float, 3>, Ponca::DistWeightFunc<PointPositionNormal<float, 3>, Ponca::SmoothWeightKernel<float> >, void>'
291 Check = Base::PROVIDES_ALGEBRAIC_SPHERE,
292 ~~~~~~^
293 \endcode
294 The capability `PROVIDES_GLS_PARAMETRIZATION` tells that GLSParam provides the GLS parameterization \cite Mellado:2012:GLS, and other components can access the related information.
295
296 In order to ease tools combinations, each class declare a set of *required* and *provided* capabilities, which are detailed in the Section \ref concepts_provides.
297
298 \note In its current version, the requirement/capability system offer limited protection over the combination of tools providing the same capabilities.
299 Indeed, it is possible to build ill-formed combinations where two computations attempt to save their results at the same place, e.g. when using two fitting technique for the same primitive:
300 \snippet fit_plane.cpp Conflicting type
301 This case is not detected at compile time, but rather dynamically when calling finalize(), which returns `CONFLICT_ERROR_FOUND`.
302 Internally, this is implemented by checking if the primitive is already valid (ie. it has been computed already) when finalizing the computation.
303 This limitation is expected to be resolved in upcoming releases.

305 \subsubsection fitting_cast Providing cast operations
306
307 As explained earlier, the class hierarchy can not reach every function of every through the provided `Base` type due to overloading.
308 The mecanism Ponca implements is a cast operator. Fortunately, this is quite common and a macro helps to define them. Add the following
309 within a public section of your class:
310
311 \code
312 PONCA_EXPLICIT_CAST_OPERATORS(ClassName, className);
313 \encode
314
315 Where the first parameter is the class name and the second is the name of the cast function which is, most of the time, the same as the
316 class name excpet a first lower case letter. Note that this macro requires the template parameter to be named following ponca conventions.

318 \subsubsection fitting_newdiff New primitive differentiation (BasketDiff API)
319
320 The API for defining new estimator that differentiate the primitive follows the same principle from classical estimator. The base class is BasketDiff which follows the
321 same CRTP / inheritance principle except that a Basket is required as a first parameter.
322
323 \snippet concepts.hpp ComputationalDerivativesConcept
324
325 \note PrimitiveDer is the default entry point to most classes used in BasketDiff
326
327 A few addition should be notted however. First, the class is now templated on a flag (an integer) representing the variables according to which we want to differentiate. For now,
328 Ponca defines only two set of variables (&lt;Fitting/enums.h&gt;):
329
330 - FitScaleDer: The derivatives of the primitive parameters w.r.t the scale (the neig radius)
331 - FitSpaceDer: The derivatives of the primitive parameters w.r.t the position of evaluation
332 - FitScaleSpaceDer: Compute derivative for both types.
333

335 they are fixed-size array whose size is adapted to the total number of derivatives. Their layout is described below depending on the type of differentiation.

363 \subsection fitting_multiprimitive Fitting with multiple primitives
364 In most cases, only one primitive is included in the Basket, and it is recommended to use the helper classes provided by Ponca for fitting, e.g. CovariancePlaneFitImpl, OrientedSphereFitImpl.
365 However, Ponca also allows to combine multiple primitives, for instance to share intermediate computation results.
366 Using this functionality requires to really understand how the different primitives and fitting techniques are working.
367 In any case, it is recommended to explicitly define the computational arrangement, e.g.:
368 \snippet basket.cpp HybridType
369
370 After fitting, this object provides access to both the plane and the sphere, through the respective cast operators #AlgebraicSphere::algebraicSphere() and #Plane::plane().
371
372 \section fitting_cuda Cuda
373 Ponca can be used directly on GPU, thanks to several mechanisms:
374 - Eigen Cuda capabilities, see <a href="http://eigen.tuxfamily.org/dox-devel/TopicCUDA.html" target="_blank">Eigen documentation</a> for more details.
375 You *need* to use a consistent `Eigen::Index` on both CPU and GPU if you plan to transfer memory between the computing units.
376 That's why we recommend to set the following preprocessor variable when compiling your project:
377 \code -DEIGEN_DEFAULT_DENSE_INDEX_TYPE=int\endcode
378 You might also need to define the `--expt-relaxed-constexpr` preprocessor option for `NVCC`.
379 Example of working cmake file (see \ref example_cu_ssc_page):
380 \code
381 enable_language(CUDA)
382 add_executable(ponca_ssgls "ponca_ssgls.cu")
383 target_compile_options(ponca_ssgls PRIVATE --expt-relaxed-constexpr)
384 \endcode
385 - Automatic CPU/GPU compilation qualifiers. We use the macro
386 \code
387 PONCA_MULTIARCH void function();
388 \endcode
389 to use the same code for C++ and CUDA. It has no effect when the code is compiled with GCC or Clang, but it will force the compilation for both host and device architectures when compiling with nvcc. A similar macro system is provided for mathematical functions, to switch between STL and CUDA versions.
390
391 Check the \ref cu_ssgl_intro_sec "C++/Cuda" and \ref example_python_ssc_page "Python/Cuda" (using PyCuda) examples for more details and how-to.
